
A **bug report** (also called defect report) documents the issue clearly so developers can reproduce and fix it.

# Bug Report Template

Typical fields in a bug report:

1. **Bug ID** – Unique identifier (e.g., BUG-001)
    
2. **Title / Summary** – Short, clear description of the bug
    
3. **Description** – Detailed explanation of the issue
    
4. **Steps to Reproduce** – Step-by-step actions to trigger the bug
    
5. **Expected Result** – What should have happened
    
6. **Actual Result** – What actually happened
    
7. **Severity** – How badly it impacts the system (Critical, High, Medium, Low)
    
8. **Priority** – How urgently it should be fixed (High, Medium, Low)
    
9. **Environment** – OS, browser, app version, test environment (QA, Staging, Prod)
    
10. **Attachments** – Screenshots, logs, or screen recordings (if possible)
    
11. **Status** – Open, In Progress, Fixed, Retest, Closed


# Example Bug Report

**Bug ID:** BUG-001  
**Title:** Login fails with valid credentials  
**Description:** 
+ User is unable to log in even with correct username and password. 
+ System shows an error message.  
**Steps to Reproduce:**
1. Navigate to the login page
2. Enter valid username and password
3. Click on "Login" button
**Expected Result:** User should be logged in and redirected to the dashboard.  
**Actual Result:** Error message appears: _"Invalid credentials"_.  
**Severity:** High  
**Priority:** High  
**Environment:** Windows 11, Chrome v115, QA server  
**Attachments:** Screenshot of error message  
**Status:** Open
